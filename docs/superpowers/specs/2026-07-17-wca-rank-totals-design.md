# WCA Rank Totals Design

## Status

Approved for implementation on 2026-07-17.

## Summary

Create a small public repository, `peterish8/wca-rank-totals`, that uses GitHub Actions and the official WCA Results Export to publish exact per-event ranked-competitor totals. The generator will not use SQL, MySQL, Supabase, or a continuously running server.

Cubify will continue to obtain a competitor's personal records from the official WCA person API. It will replace its current per-event requests to the unofficial full-ranking JSON files with one request to the compact generated `data/rank-totals.json` file.

## Goals

- Count people with valid official WCA rankings, not merely registered people.
- Produce totals for every event, separately for single and average.
- Produce world, continent, and country totals for each event and rank type.
- Treat each current WCA ID as one competitor even if historical `persons.sub_id` records exist.
- Use only official WCA export data as the source of truth.
- Run entirely on a standard public GitHub Actions runner.
- Publish a compact, deterministic JSON file that Cubify fetches once.
- Preserve the last known-good output when an update fails.

## Non-goals

- Reproduce the complete unofficial WCA REST API.
- Publish full ranking lists or individual result histories.
- Count registrations or people without a valid ranked result.
- Maintain a database or a continuously running API server.
- Derive daily changes from an unavailable official delta feed.
- Make Cubify's comparison page depend on the totals file; it only compares ranks.

## Source of truth and counting semantics

The generator uses the WCA Results Export v2 metadata endpoint and these TSV members:

- `WCA_export_countries.tsv`
- `WCA_export_persons.tsv`
- `WCA_export_ranks_single.tsv`
- `WCA_export_ranks_average.tsv`

The rank tables are authoritative because each row represents a competitor's official best result for one event and rank type. A ranked competitor is counted when all of the following hold:

1. A rank row exists for the competitor and event.
2. `world_rank`, `continent_rank`, and `country_rank` are positive.
3. The WCA ID resolves to exactly one current person row where `sub_id = 1`.
4. The current person's country resolves to exactly one country and continent.
5. The WCA ID has not already appeared for that event and rank type.

A competitor can therefore count toward single but not average for the same event. A person with no rank row is never counted. Historical person rows with `sub_id > 1` are ignored for geographic grouping and cannot multiply the count.

Rank numbers themselves are not used as totals. This is important because ties can make the largest rank differ from the number of ranked competitors. The generator counts unique rank rows instead.

## Repository boundaries

### `peterish8/wca-rank-totals`

Owns downloading, parsing, validating, and publishing official aggregate totals.

Proposed structure:

```text

```

The implementation will use pinned Python 3 and the standard library. Runtime package installation is not required.

### `peterish8/Cubify`

Owns fetching a player, looking up the corresponding totals, calculating the displayed top percentage, and presenting source freshness.

The two repositories communicate only through the versioned JSON contract below.

## Update workflow

The GitHub Actions workflow supports `schedule` and `workflow_dispatch`. Its scheduled trigger is `17 6 * * *` (06:17 UTC daily), avoiding the busy start of an hour while leaving time for the WCA's normal export process.

1. Request `https://www.worldcubeassociation.org/api/v0/export/public`.
2. Validate the metadata fields and require export format major version `2`.
3. Compare `export_date` with the committed JSON's `source.exportDate`.
4. Exit successfully without downloading the archive when the dates match.
5. When the export is new, obtain the four required TSV members.
6. Build current-person and country mappings.
7. Stream the single and average rank tables and update counters.
8. Run all structural and aggregate validations.
9. Serialize the result deterministically to a temporary file.
10. Replace `data/rank-totals.json` only after validation passes.
11. Commit only when the generated bytes differ from the committed file.

A manual run remains available. Workflow concurrency will prevent two publishers from running simultaneously.

The job needs only `contents: write`; it requires no stored secrets beyond GitHub's scoped `GITHUB_TOKEN`. The generated data is committed to Git rather than uploaded as a GitHub Actions artifact.

## Selective ZIP acquisition

The official WCA TSV download is a complete ZIP archive. The downloader will first attempt selective member retrieval:

1. Use HTTP byte-range requests to read the ZIP central directory.
2. Locate the four exact required member names.
3. Fetch only their compressed byte ranges.
4. Decompress and stream them without extracting arbitrary paths.

This optimization is feature-detected. The downloader will automatically fall back to downloading the complete ZIP when the server does not return valid range responses or the ZIP layout is unsupported. Both paths produce identical parser inputs and are tested against the same fixtures.

The implementation must not claim a performance target until both paths are benchmarked against a real WCA export. Correctness takes precedence over selective downloading.

## Processing algorithm

### Phase 1: countries

Build a small mapping:

```text
country_id -> { iso2, continent_id }
```

Country keys in the output are uppercase WCA `iso2` values. Continent keys retain the WCA's canonical IDs, including a leading underscore when present, so Cubify can use the official person API value directly without a translation table.

### Phase 2: current persons

Stream the persons TSV. Retain only `sub_id = 1`:

```text
wca_id -> { country_iso2, continent_id }
```

Two current rows for one WCA ID, an unknown country, or a country without a continent is a hard validation failure.

### Phase 3: ranks

Process `ranks_single` and `ranks_average` independently. For each valid row:

- Verify the person mapping exists.
- Verify the three official ranks are positive.
- Verify `(event_id, person_id)` is unique within the rank type.
- Increment the event's world counter.
- Increment the current continent counter.
- Increment the current country counter.

The parser streams rows and does not load either entire TSV table into memory. It retains only the person mapping, aggregate counters, and uniqueness keys required for correctness.

## Published JSON contract

The initial schema version is `1`:

```json
{
  "schemaVersion": 1,
  "source": {
    "name": "World Cube Association Results Export",
    "exportDate": "2026-07-17T03:24:49+00:00",
    "exportFormatVersion": "2.0.2",
    "archiveUrl": "https://exports.worldcubeassociation.org/results/example.tsv.zip",
    "url": "https://www.worldcubeassociation.org/export/results",
    "attribution": "Based on competition results owned and maintained by the World Cube Association."
  },
  "events": {
    "333": {
      "single": {
        "world": 1,
        "continents": {
          "_Asia": 1
        },
        "countries": {
          "IN": 1
        }
      },
      "average": {
        "world": 1,
        "continents": {
          "_Asia": 1
        },
        "countries": {
          "IN": 1
        }
      }
    }
  }
}
```

Serialization rules:

- Event IDs, continent IDs, and country IDs are sorted lexicographically.
- Numbers are non-negative integers.
- Missing rank types or regions are omitted rather than represented by guessed values.
- Build time is not included, keeping output deterministic for a given export.
- `source.exportDate` is the freshness marker used by the workflow and Cubify.
- `source.archiveUrl` records the versioned official TSV archive used for reproducibility.
- A future breaking contract requires a new `schemaVersion`.

## Validation and last-known-good behavior

Before publication, the generator verifies:

- All four required TSV members exist and have their expected headers.
- Exactly one current person row exists for each retained WCA ID.
- Every ranked WCA ID has a current person and geographic mapping.
- Every rank triple is positive.
- No duplicate `(event_id, person_id)` occurs within a rank type.
- For every event and rank type, country totals sum to the world total.
- For every event and rank type, continent totals sum to the world total.
- The JSON satisfies the schema and can be parsed again.
- Reprocessing the same fixture produces byte-identical output.

A decrease from the previous export is logged but is not automatically rejected because the WCA may correct official results. Structural inconsistency, rather than an arbitrary percentage threshold, determines failure.

Generation occurs in a temporary location. Any download, parse, or validation failure exits the action before the published file is replaced or committed. The previous commit therefore remains the last known-good dataset.

## Cubify integration

Cubify's player lookup remains:

```text
GET https://www.worldcubeassociation.org/api/v0/persons/{WCA_ID}
```

For each search, Cubify will fetch the following file once and reuse it for every event:

```text
https://raw.githubusercontent.com/peterish8/wca-rank-totals/main/data/rank-totals.json
```

It will look up:

- `events[eventId][rankType].world`
- `events[eventId][rankType].continents[player.country.continent_id]`
- `events[eventId][rankType].countries[player.country.iso2]`

This replaces up to six unofficial rank-list requests per event with one compact totals request for the whole page.

The displayed top share is calculated as:

```text
topPercent = rank / totalRankedCompetitors * 100
```

The current `(rank - 1) / total` display calculation will be replaced because it reports rank 1 as top 0% and never reports the bottom-ranked person as 100%.

The UI will format values below `0.1` as `Top <0.1%` rather than rounding them to `Top 0.0%`. Other values retain enough decimal precision to avoid materially changing the category shown.

If the event, region, or total is absent, Cubify shows the rank without a percentage instead of substituting zero. If a live official rank is greater than the export total, Cubify treats the aggregate as temporarily stale and hides that percentage until the next successful data update. The UI will expose the WCA export date and attribution.

## Tests

Unit fixtures will cover:

- A WCA ID with `sub_id = 1` and historical `sub_id = 2`, counted once using the current country.
- A registered/current person with no rank rows, counted zero times.
- A competitor with a valid single but no average, counted only for single.
- Tied rank values, proving totals come from unique people rather than the maximum rank number.
- Duplicate rank rows, which must fail.
- Missing current-person or country mappings, which must fail.
- Invalid or zero rank fields, which must fail.
- Country and continent reconciliation.
- Deterministic JSON ordering.
- Successful selective ZIP reads and automatic full-download fallback.

The GitHub Actions workflow will run the full unit suite before generating or publishing data. A generation run is also an integration test against the current official WCA export.

## Security and operational constraints

- Process only four exact ZIP member names; never extract arbitrary archive paths.
- Validate metadata, headers, content lengths, and decompression results.
- Use HTTPS official WCA URLs obtained from the official metadata endpoint.
- Pin third-party GitHub Actions to immutable commit SHAs where practical.
- Give the workflow only `contents: write` permission.
- Do not use repository or organization secrets.
- Do not upload the full WCA archive or extracted TSV files as artifacts.
- Remove temporary downloads at job completion.
- Document that scheduled workflows on inactive public forks may need to be manually re-enabled.

## Success criteria

The design is successful when:

1. A clean GitHub Actions run generates totals from the current official WCA v2 export without SQL.
2. The historical-person fixture that duplicates the old unofficial API count produces exactly one competitor.
3. Every published event/type passes world-country-continent reconciliation.
4. A second run against the same export exits without changing Git.
5. Cubify performs one totals request per search and displays correct top percentages.
6. A failed update leaves the previous JSON available.
7. The measured workflow runtime and transferred bytes are recorded in the README after the first production run.

## Rollout

1. Create the public `peterish8/wca-rank-totals` repository.
2. Implement fixtures, parser, validation, downloader, and deterministic serializer.
3. Run locally against fixtures, then against the current official export.
4. Enable and manually run the GitHub Actions workflow.
5. Inspect the generated totals and workflow benchmark.
6. Integrate Cubify with the compact file and corrected percentage formula.
7. Build and smoke-test Cubify with multiple known WCA IDs and regions.
8. Push Cubify and verify its production deployment.
9. Optionally submit the independent `sub_id = 1` bug fix upstream to `robiningelbrecht/wca-rest-api`; Cubify will no longer depend on that project for totals.
